谈谈高效能的工程师

提到高效能，很多人脑海中可能会浮现出以下场景：

  • 繁忙而快节奏的投行办公室里，主管面对着一件又一件烦杂的事情，总是专注而快速的做出了准确的决策。

  • 昏暗的房间内，黑客极度专注，键盘被高速的敲击着。没过几分钟，黑客停了下来，轻舒一口气，又完成了一个任务。

相信大家也都希望自己能具备如此高效能的能力，把活高效、高质量的完成，下班后能好好的休息、充电，享受生活。

针对效能提升方面，市面上已有了非常多的书籍和研究资料，涵盖了不同行业、不同场景的方方面面。 本文则是希望针对工程师的高效能方面，能较为系统的梳理和综合多方因素，给作为工程师的我们带来一些帮助。

# 如何定义高效能

高效能的定义，市面上有很多不同角度的不同定义，以下是我从工程师的角度综合得出的一个定义：

“能通过合适的方式高质量、高效率的完成目标。”

此定义强调了高效能最关键的2个影响因素：效率， 质量。 其中效率可以通过“产出 / 单位时间”得到衡量，而质量则可以通过“成果品质与预期值的比对”来评估。

# 如何做到高效能

把事情做到高效能，主要包括“有效”，和“效率”两个方面。

# 1. 有效（effective）

有效是指把事情做对、做好。事情的目标方向是正确的、且事情完成的质量是高的。

# 1.1 把事做对

在《人月神话》中有提到两个非常重要的概念：本质复杂度（Essential Complexity）和偶然复杂度（Accident Complexity）。

本质复杂度是解决一个问题时，无论怎么做都必须要做的事；而偶然复杂度则是因为选用的做事方法不当，例如选错方法或工具而导致要多做的事。工程师要想做对事，必须要尽可能减少事情的偶然复杂度。以下的几个关键原则对减少偶然复杂度十分有帮助。

# 以终为始

制定目标、并让目标更加清楚、正确。其中一个十分有效的去实践以终为始原则的技巧是：

尝试倒着考虑事情。

想像我们有一个大学生弟弟，目前大四行将毕业，正面临着是选择考研还是直接去找工作。这时，他可以尝试去倒着做分析。

首先，这名大学生需要考虑清楚的他自己理想的职业目标。比如是在大公司成为非常资深的软件开发专家，抑或是综合能力突出的技术管理人才，还是说更想要对某方面的课题做深入的研究，争取在研究上取得突破，等等。

如果他希望是成为综合能力突出的技术管理人才，那么3年的研究生可能就不适合他了，实际工作中的技术、业务、沟通等方面的事情处理的经验将对他实现成为技术管理人才的目标更有帮助。

确定好自己的选择后，他下一步就需要去准备找工作的基本要求，学习目标工作岗位的知识技能了。

再举另外一个倒着分析的例子，做旅行计划。

首先，我们从旅行结束时所期望的结果开始考虑，也就是能够按时顺利的回程。这就意味着我们需要订到合适时间的回程票。

紧接着，这次旅行结束时，我们期望自己本次旅行的主要目标是达到的，比如说这趟旅行一定需要去的几个地方，一定要做的几件特别的事情。那么我们围绕这些旅行的主要目标去做相应的规划就更加心里有数了。

规划过程中，考虑到需要的回程时间，需要满足旅行主要目标的行程安排，以及其他的考虑因素后，我们基本就可以倒推出理想的出发日期了。此时，我们就可以踏实的提前去请年假、买机票、订酒店、买门票、公交票、租车等等。（这些事中也可能有变化，比如项目太忙无法请到理想时间的年假，那就需要再相应进行调整）

在实践以终为始的过程中，一个经常会遇到的问题是，如何把目标考虑清楚。有些目标相对比较单纯，涉及的方面较少，通过自己的分析和考虑基本可以将目标缕清楚；然而有些目标相对复杂，单单以自己的分析未必足以将目标考虑的清楚全面。以下几个原则有助于将目标考虑的更清楚:

  1. 提问。不懂就问是最朴素但是非常有效的一个原则。有疑问去问，去确认能够有效把事情了解的更清楚。

  2. 定义清楚目标完成的标准（Definition of Done），与相关人员进行沟通确认，将目标尽可能细化成符合SMART标准的内容。

  3. 扩大自己工作的上下文，从更高层面的视角考虑。

    举个例子，当我们接到一个大的业务需求，想了解清楚这个需求的目标。我们就可以向对于这个需求更加了解的人，比如产品经理去提问和沟通。交流过程中力求将需求完成的标准（DoD）了解清楚，此外可以将自己的思路从更高层面的视角，比如产品、公司、市场的视角去考虑，从而更加全面的了解需求目标。我们可以问以下这些与需求目标相关的问题：为什么要做这个需求？目标用户会在什么场景用这个需求的功能？会如何使用？达成需求的目的还有其他更好的解决方式吗？如何衡量这个需求功能的有效性？

总结：以终为始是一个将目标考虑得更清楚的持续的过程，这里需要做好计划与执行的平衡。比如在做需求开发的时候多想想：”做这个的目的是什么？“。

# 任务推演和分解

对目标的计划在以终为始的基础上进行一步步的目标进行分析和拆解。

一个软件开发工程实践中非常典型的例子是测试驱动开发（TDD）。TDD会将开发任务逐步拆解到测试用例的维度，通过拆解将所需要完成的目标进行细化，具体化，大目标拆成可执行的小目标。TDD主要的步骤包括：

  • 将开发任务拆解到几个独立子任务。
  • 各子任务拆解出各独立的功能。
  • 各功能继续拆解，拆解到不同场景下的”单元“，也就是测试用例。

然而有许多目标的拆解过程并不容易，需要我们先去做比较详尽的分析和推演。

Elon Musk对于宏大目标的推演和拆解上就做得非常厉害。他多次提到自己的人生目标之一是：有生之年让火星成为人类的第二家园。

为此，他将这一目标分解成了2部分：人类怎么住火星？以及人类怎么去火星？

以下是我对于马斯克对于这2个问题大致估计的一个推演链路：

人类怎么住火星？

  1. 短期方案：建立地面或地下的封闭式人类活动城市站；
  2. 长期目标：让火星气候适合人居住
  3. → 通过加热南北极（方案未确定，提出的有反射太阳光的镜子、核爆）的手段实现火星的温室效应。
  4. → 需要供给水、空气、能源、食物等，以及地下空间
  5. → 需要火星有相应的供给设施、以及火星城市构建工程所需的工程设备→ 需要火箭把这些东西都运过去
  6. → 火箭得够大 + 这些设施最好能拆解
  7. → 造 SpaceX Starship 超级火箭！

人类怎么去火星？

  1. 造 SpaceX Starship 超级火箭！一次可运输1000多人；但是超级贵，大部分人付不起怎么办？
  2. → 降低运输成本：可回收火箭、太空中补充燃料
  3. → 可回收火箭、太空中补充燃料怎么实现?

以及另外一个非常关键的问题：这里所需的大量资金怎么办？

  1. Starlink WiFi项目的收入
  2. StarShip火箭用于地球民用交通工具的收入
  3. StarShip火箭用于私人登月服务的收入
  4. NASA的投资
  5. Telsa的盈利支持？SpaceX、Starlink、neuralink上市？其他？

从这里我们大致上也能理解马斯克目前所做的这么多事业背后可能的目标了。

# 1.2 把事做好

工程师想要把事情做好，必不可少的是专业意识、和合作意识。

专业意识（Professional）

良好的专业意识包括但不仅限于：

  1. 功能的范围和场景考虑全；
  2. 代码尽早重构（@TDD），避免写出可读性、可维护性差的代码；
  3. 参与Code review，改善自己的代码不足，发现队友的代码问题。
  4. 持续的、深入的学习和实践专业技能和知识，能考虑到优质方案解决问题。

合作意识

事情要想做好，光凭单干是绝对不够的。 我们可以通过沟通反馈，双赢思维，知彼解己，累积信任等等原则来改进合作。这里推荐《高效能人士的七个习惯》这本书，其中对于这些原则有很好的阐释。

# 2. 效率（efficient）

效率是指通过高效率的方法、工具快速把事情做完。这里同时要保证事情完成的是有效的。

# 效率工具、功能

俗话说，工欲善其事必先利其器。效率工具、效率功能是工程师必不可少的提效工具。以下是收集到的一些好的工具：

  • 一台好电脑，双屏幕显示器。
  • 本地IDE（VS Code等）配置好（包括插件、快捷键、主题等）：
    1. 参考其他人分享的设定：https://github.com/CodingGarden/vscode-settings
    2. 自行搜寻需要的插件、设定。
  • Mac应用、工具网站：
    1. Alfred 搜索
    2. Spectacle窗口宽高快速切换
    3. F.lux护眼屏保
    4. 找APIs做测试数据或demo用？: https://github.com/public-apis/public-apis
    5. 不想装photoshop但要用其中的功能？：https://pixlr.com/，https://www.photopea.com/
    6. 给函数、变量命名想不到精辟的表达词？：https://www.thesaurus.com/ 同义词查询
    7. 想管理、记录、跟踪一些工作、生活的信息：Notion: https://www.notion.so/ , Trello, etc.
    8. 文档格式转换：Google一下吧
  • Chrome插件：
    1. 自行搜寻需要的插件，这边不详细介绍了。
  • Terminal：
    1. bash、zsh环境的配置，包括plugin，alias的配置
    2. vim。

# 应用程序提供的效率功能

  1. 模糊搜索。Google、以及各网站的搜索功能；VS Code通过搜索开启文件，搜索查找内容；
  2. 快捷键。MacOS，Chrome，VS Code，Terminal，以及其他应用程序中都有提供，应善加利用来提高效率。
  3. 整合。例如Notion，将笔记，Todo list, 看板，计划，日历全都整合到一起，提高整个工作管理的效率。
  4. 偏专业性的效率功能。例如JIRA中的甘特图，Issue的Type，Tag的分类管理等
  5. 自动化：
    1. 原则：谨慎做，避免因自动化带来功能和质量的风险
    2. 工作流程可以尽可能自动化，如持续集成、持续部署
    3. 这部分可以独立做专题，就不展开说了
  6. 软件设计：代码功能的封装、复用
    1. DRY（don't repeat yourself）：不仅是工作量的浪费，还会大大降低代码的质量和可维护性。
    2. 封装库由专人维护，其使用者不需考虑库内部的逻辑，减少了开发工作量
    3. 业绩最佳实践的沉淀：各类技术以及不同场景下的设计模式、架构方法；优秀的开源公共库、框架等

# 时间管理

  1. 要事第一的原则：优先级、事情的紧急度/重要度的管理
  2. 番茄工作法：劳逸结合 - 劳与逸都投入 （《番茄工作法图解》：https://book.douban.com/subject/26099295/） ；可以下载对应的网站或App。
  3. 利用deadline来驱动自己高效做事，可以分解成多个小的deadline，一个个推进。也可以利用外部的压力来推动自己高效的行动起来。

# 如何持续高效输出

高效能的工程师还需要保证其可持续性。这里就包含2个非常关键的前提：想把事情做好，以及能把事情做好的基础。

# 动机 - 想把事情做好

  1. 积极主动，动力和激情。作出承诺与信守诺言。人有：自我意识、想象空间、道德心和自由意志。注意力放在影响圈而非关注圈，从而扩大影响圈。
  2. 兴趣：参与其中的本身就很享受。
  3. 工程师文化、工匠精神：探索、创作的欲望和坚持！

# 本钱 - 能把事情做好

工程师健康的身体、充沛的精神是能把事情做好的基础：

  1. 经常锻炼身体
  2. 良好的作息，休息
  3. 健康的饮食
  4. 快乐、正向的心态
  5. 咖啡，红牛等等

# 结尾

希望本文所谈到的高效能的内容能够有所帮助，大家一起努力成为更加高效能的工程师！
