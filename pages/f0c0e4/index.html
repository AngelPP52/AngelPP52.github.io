基于Spark的数据比对与监控系统

# 背景

在业务系统的日常运行中，势必会面对大量业务系统间数据同步的场景，以及相应的数据问题。

通常会有几种场景：

  1. 同步基础数据进行关联查询、扩展列

    如：SKU数据

    可能的数据问题：列的值不一致；行记录缺失；

  2. 业务系统之间存在关联的状态流转

    如：采供系统的预入库单 与 仓库系统的入库单 之间存在状态关联

    可能的数据问题：两个系统存储的状态不符合业务约束；

同步机制可能是调用接口、队列 / GDS等，尽管有重试、补偿、记录错误等机制去保证数据最终一致性，但是仍然有可能出现数据问题。

由于数据分布在不同业务系统、由不同的团队进行管理，因此不容易发现问题、可能要等用户上报数据问题时才能发现。

因此，我们需要一套持续的监控机制主动发现这些数据问题。


# 解决方案

# 解决思路

如果在微服务架构下解决以上需求，会遇到两个方面的问题：

1.访问限制：无论通过接口或访问数据库，都对现有的访问限制体系造成很大挑战；

2.占用在线资源：可能对在线服务的稳定性造成影响；

通过分析，以上场景其实离线简单的数据处理场景，因此选取基于公司的 Airflow/Spark/Presto 来搭建持续的数据比对监控系统。


# 主要组件及概念

# Airflow

调度和监控工作流的平台，基于Python 编写的有向无环图（DAG）进行任务调度和管理。 主要用于大数据生态的组件，也可以作为crontab的升级版应用于运维操作。

# Task

Task即任务，是Airflow调度的最小单位；通过Operator来创建任务。

# Operator

Operator是一系列封装好的常用TaskObject，如：

  • SparkOperator：提交一个Spark任务

  • PythonOperator：执行一个Python Function

  • SSHOperator：通过SSH连接主机并执行Shell命令

# DAG（Direct Acyclic Graph）

定义一组Task集合，使用Python描述了一系列Task及其依赖关系、调度逻辑。

# Sensor

检查预设的条件是否满足，满足后执行下游的Task。

# SLA（Service Level Agreements）

一段时间内应该完全的操作，否则触发另外任务，如发邮件发短信等；在 DAG-Level 和 Task-Level 都可以进行定义。


# Spark

批处理与实时运算的计算框架。

# YARN

Hadoop 2.0的资源管理系统。通过一个全局的资源管理器ResourceManager和每个应用程序独立的ApplicationMaster进行资源调度与管理。

ResourceManager负责整个系统的资源管理和分配，可以通过Queue进行业务线的资源限制与隔离。

ApplicationMaster负责单个应用程序的管理。

# Spark Driver Server/Client Mode/Cluster Mode

  • Spark Driver Server：一台业务组共享的服务器，用于提交Spark任务、存放任务代码、暂时存放输出结果。

  • Client Mode：以Client Mode提交时，Spark Driver Program运行在Client上（Spark Driver Server），结果与文件存放在Client上，使用Client上的运行环境、存储和计算资源。

  • Cluster Mode：以Cluster Mode提交时，Spark Driver Program运行在由YARN分配的ApplicationMaster上，结果与文件存放在Cluster上某一节点（Node）。

# Container/Executor/Job(Task)

  • Yarn Container：Spark Executor的运行环境容器

  • Spark Executor：Spark任务执行者，相当于进程

  • Spark Job（Task）：Spark任务，最小的执行单位


# Presto

分布式SQL查询引擎


# 系统架构

数据集成数据流

  • 基于Data Ingestion系统设置ETL任务每天同步数据到Presto

数据比对与监控数据流

  1. SSH Operator：Spark Driver Server上执行Spark-submit
  2. Spark-submit：以Client Mode提交Spark Job
  3. Spark Job：通过SQL读取Presto数据并进行数据分析
  4. Spark Job：整理结果写入MySQL
  5. SSH Operator：执行Spark Driver Server上执行统计结果的Python脚本
  6. Python script：读取监控结果数据整理并发送 监控结果 的SeaTalk告警

# 问题与思考

# 数据的实时性

由于数据集成的数据每天执行一次ETL任务、不同数据源执行任务的时间不一致，引入了额外导致数据不一致的因素。

所以，数据比对系统主要用于监控变动不频繁的数据、或状态流转的终态。

如果需要想基于Spark体系解决这个问题，有两方面的思路：

  • 基于Airflow提供的Hooks或Sensor，接受ETL任务的触发

  • 基于Spark Streaming实时计算，实时监控数据

# Spark的定位与使用场景

BE的日常工作主要是基于微服务架构处理在线业务场景，但离线场景的计算、数据处理/分析、监控缺少一些工具；而Spark在这些方面作为补充。

同时我们应该注意到，数据对比是辅助的、滞后的“兜底”策略，首要工作是提升在系统中数据同步的可靠性。
