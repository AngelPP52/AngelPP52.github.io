            # django-models-documentor 简介

            django-models-documentor 是我们在供应链内部为 Django 项目开发的用于 model 层生成静态文档的工具，方便了开发童鞋在编写代码中一并地完成了文档的编写以及将文档进行共享。

            # 背景介绍

            Shopee PMS 系统是提供给运营团队使用的采供管理系统，作为一个审批与决策系统，需要与数据分析团队紧密配合，围绕在 PMS 系统周围大致的协作流程如下图所示：

            如图中红色沟通路径所示，PMS 研发人员与 BI 团队分处深圳和新加坡或者业务运营当地，跨国团队的沟通成本本身较大。而在日常的这个沟通上，还容易发现以下问题：

            • 信息滞后：随着系统研发迭代，原来BI同事了解到的信息，可能变成了错误的信息；
            • 沟通的重复：由于没有统一的信息载体，对于同一个业务的设计，不同地区的 BI 团队成员可能重复过来咨询，对于 PMS 研发人员是一种重复的负担；
            • 内部沟通协调成本：PMS 研发人员内部对不同业务模块的系统设计了解不尽相同，如果遇上 BI 同事来了解某个模块的系统设计，还必须在研发团队内部找到了解该模块的同事。

            于是，为了解决上面的问题，提升整体协作效率，我们需要有一个统一的数据库设计的信息管理工具。

            # 早期的方案：基于 Google Sheets 的数据字典

            最开始，为了快速满足 BI 团队的需要，我们针对高频查询分析的数据库表的设计做了数据库设计的数据字典，并且由开发人员手动填写到 Google Sheets，并分享链接给 BI 团队，以此将一个点对点的沟通问题改为一个点对面的沟通问题，沟通效率确实能够明显提升。

            (图示：基于 Google Sheets 共享的方案)

            (示例图：早期我们手工整理的数据字典示例)

            可以看到，这个方案的话，在一定的前提下，基本就解决 BI 团队查阅数据库信息的问题了，所以在上面新的协作流程里，BI 团队到 Google Sheets 文档的信息路径不再标红。尽管这套机制平稳运作了挺长一段时间，我们还是认为这个方案仍旧不够好，为什么呢？还是得回到这个方案刚才没有说的“前提”上，这些前提就是问题所在：

            • 研发团队成员必须遵守比较强的规范才能确保 google sheets 上的数据字典的信息是足够有效的，这些规范主要包括：
              • 必须及时整理更新最新的数据库设计信息，这个及时的边界可以视为新的功能发布上线的时间
              • 必须保证代码逻辑与文档一致，如果不一致，文档就是错误的引导，比没有文档可能更糟糕
              • 必须遵守统一的格式，如果格式或者表述方式不够统一，将会造成文档查阅者的困惑
            • 研发团队除了编写代码，还需要编写文档，代码与文档分离，在开发团队看来，更像是重复工作

            于是，我们也在思考：**有没有一种方式或者工具，可以让我们在写代码的时候，顺便就把文档也写好？**这样，只要代码发布，文档自然就是最新的，且代码与文档同行，更重要的是，比起打开浏览器访问 Google Sheets 然后去对数据字典做填空这种枯燥的工作，我们更乐于在代码中去编写我们的文档，因为这都是顺手的事情，没有需要刻意记得去另一个地方更新文档的心智负担。

            # 文档工具的选型与设计

            # 设计目标

            经过上面的思考，我们的目标很简单：选择或者自行开发一款脚本工具，用于替代 Google Sheets 表格的方式，让开发者在开发阶段就可以顺手编写设计文档，并支持自动部署到内部办公人员可访问的空间。

            而结合已有的数组字典的格式，我们对待选型或者开发的工具的功能性需求为：

            • 支持编写的信息结构
              • 数据表的名称、表的使用场景以及表所属业务模块
              • 数据表的列列表，包含：
                • 列名
                • 列的含义
                • 列值的枚举
                • 列的文档说明，可以用来描述业务逻辑等非标准格式的信息
            • 支持一种轻量级展示设计内容的方式，比如静态网页，或者自动导出到公共的 Google Sheet

            # 已有的工具选型 —— The Django admin documentation generator

            官方链接：The Django admin documentation generator

            我们调研了 Django 官方的这个生成工具，但是由于以下原因，我们没有直接采用：

            • Django admin documentation generator 需要基于 Django Admin 运作，但是我们并不启用 Django Admin
            • Django admin documentation generator 是作为一个 Django app 运行的，意味着需要提供一个运行时资源，且文档服务本身会和系统服务耦合在一起
            • Django admin documentation generator 要求 Django 项目代码的结构符合标准的规范（installed app），特别是要求所有 model 都继承 django.db.models.Model，而我们实际项目里，由于分表的场景，我们有些 Model 并不直接继承自 Django 的 django.db.models.Model
            • 我们的实际项目由于已经有一些自己实现的扩展，并不能完全契合 Django admin documentation generator 的工作场景

            # 设计思路

            虽然我们没有直接采用 Django admin documentation generator，但是我们依然从它身上得到了一些灵感：尽可能复用 Django admin documentation generator 的文档编写方法。这样对我们来说，有几点好处：

            • 因为对 Django 无侵入，不用担心后续和 Django 之间的维护问题
            • 可以复用 Django 已有的实现，不用再去自己造一遍文档解析的轮子，这个稍后会看到具体的说明
            • 不用去设计各种奇怪的文档语法，避免开发人员刻意学习，一切都还是 Django 本身，没有额外要学习的内容

            # 数据表层面信息描述规范的设计和实现

            针对表层面的信息，我们复用了 Django Model 本身内置的 Meta 类的几个类属性，所有这些属性可以通过 Model._meta.{attribute}的属性访问：

            • db_table：这个属性本身用于声明当前 model 对应数据表的名字，在我们生成的文档中会对应于TABLE NAME一栏的信息；
            • app_label：这个属性会被我们工具提取出来作为数据表本身所属的业务模块的标志；
            • verbose_name：这个属性本身也是 Django Model 设计用来填写人类可读的表名信息的属性，在我们的实际设计中，用于简单描述表的使用场景。

            以下是基于此设计的代码示例以及实现过程中用到的属性查询示例：

            # 数据表字段层面信息描述规范的设计和实现

            在表字段实际的详细设计上，同样因为 Django admin documentation generator 的设计，Django 的 model Field 类都支持 verbose_name 和 help_text 选项，于是我们的设计里，我们设定 model field 上 verbose_name 用于定于字段本身含义，而 help_text 字段用于编写字段本身较长的帮助文档或者业务逻辑等。而在实现上，只要通过Model._meta.fields属性便可轻松解析字段信息：

            在信息结构上，表层级的信息、列名、列的含义、以及列的文档说明都可以轻松基于 Django 内置的特性实现，这样的话，我们的工具到目前都不需要对 Django 本身做侵入等。但是遗留了一个必要的功能：针对特定字段的枚举值的说明，如何表达？

            为了不用再将上面所示的枚举定义拷贝到设计文档中，最好是在文档中通过引用某个枚举类的方式来编写文档；另一方面，考虑到 Python 语言自身支持的 doc string 特性，可以支持通过类的 __doc__ 属性访问类的文档。综合起来，我们对字段的枚举说明文档定了个语法：在Model的 docstring 部分使用 @enum {字段名}: {枚举定义的引用路径} 的格式，这也是整个工具唯一需要的一个定制。

            在实现上，这个枚举的解析是通过简单的正则匹配来完成的，以下是核心代码：

            def parse_enums(self):
	if not self.model.__doc__:
		return {}

	enums = {}
	for line in self.model.__doc__.strip().split("\n"):  # 访问了 model.__doc__ 获取 docstring，这个是前提
		line = line.strip()
		res = re.search(r"@enum\s*(\w+):\s*([\w\.]+)", line)  # 简单的语法，整个工具仅此一个语法

		if res:
			field_name, enum_name = res.groups()[0:2]
			enums[field_name] = enum_name
	return enums

            # Model 的扫描

            前面的讨论只是讲了单个 Model 的设计资料如何组织的问题，但是为了实现这个工具的目标，还有一个重要的问题：如何扫描整个项目所有的 Model 定义？一种可能的方法是直接调用 Django 内置的 django.apps.apps.get_models 方法，但是这个方法对于我们有些问题：

            • 由于历史遗留问题，我们的代码中一些陈旧的模块组织不符合 Django 的规范，且未声明为 Django installed app，这导致上述方法无法扫描到这些模块下的 model
            • 另外，因为同样前面提到的分表问题，我们对于分表的 Model，并未直接继承自 Django 的 django.db.models.Model，而是我们内部开发实现的一个 PartitionModel，这也同样导致上述方法会漏掉一些重要的 Model

            最后，我们使用了一个比较简单粗暴的方法，大致的思路是这样的：

            • 直接遍历所有模块（在 Python 中，一个 Python文件就是一个独立的模块），基于 Python 的 pkgutil.walk_packages 方法实现；
            • 对每个模块，遍历其中引用到的所有类；
            • 判断每个类是否为 model（是否为django.db.models.Model或者PartitionModel的子类），是则加入到全局的 model 列表中，否则跳过。

            由于这只是一个静态文档生成工具，上述方法虽然从算法分析角度会觉得效率低，但是整个扫描生成过程并不会有明显的等待时间，且能够很好地解决内置的 get_models 无法覆盖到所有 Model 的问题，我们便没有吹毛求疵。

            # 命令行工具与上传部署

            完成整体工具的开发实现之后，我们便进一步开始思考：如何方便使用，以及完成部署？

            答案是命令行！由于工具本身设计用于 Django 项目，所以在命令行的实现上，方案非常简单，直接基于 Django Management Command 机制实现即可，于是我们便有了 generate_models_document 命令，以下为安装了本工具的项目执行 python manage.py 后看到的本工具提供的命令行：

            [models_documentor]
    generate_models_document
            1
            2

            同时，考虑到生成的只是一个单页的静态网页，也就是只有一个 html 网页，并且我们项目内部采用了 Ceph 分布式对象存储系统，所以我们将部署简化，将 Ceph 也用成了静态站点部署的平台：将生成的静态网页上传到 Ceph，并且在命令行工具里一并集成，所以上述的 generate_models_document 命令除了生成静态网页，还会同时自动地将网页上传到存储系统，内部用户便可以通过对应的 URL 访问此网页。

            这里上传到 Ceph 还有一点值得一提的是，如果想要网页本身在浏览器中直接渲染，需要在上传文件的过程中，设置该文件的正确的 ContentType 为 text/html，否则，当你在浏览器中打开链接时，浏览器只会帮你下载此网页文件，而不是在浏览器中渲染展示，无法模拟网站的效果。

            # 自动发布

            提供命令行以及上传的好处，就是可以将文档生成上传的过程和代码合并的过程结合到一起，在 GitLab Pipeline 的基础上，我们配置了每次代码合并之后，自动触发上述命令的执行，于是，整个文档更新的流程，自然就和代码发布的流程保持同步了。

            • 设计信息的组织整理有了更强的规范：对比 Google Sheets 的形式，我们的格式更有限，有限的格式避免无限的个人风格和发散
            • 整个工具其实功能有限，但是都有效，开发成本很低，大概一天左右能开发完，但是换来的协作效率的提升，团队也不用再去应对各种临时的不必要的当面咨询

            # 后续优化方向

            • 关系的表达：目前虽然仅仅通过记录数据表自身的信息就已经满足80%甚至90%的沟通场景了，但是实体之间的关系表达也是沟通的重点
            • 提取成公开的开源项目：目前这个工具的实现偏内部定制化较高，如果可以将定制部分抽象成可配置的选项，则可以轻松实现将该工具开放给有类似场景需求的人
